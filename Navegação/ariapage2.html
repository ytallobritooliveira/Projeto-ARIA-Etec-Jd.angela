Quais são os principais atributos ARIA e como aplicá-los corretamente?

  • role: Define a função de um elemento (ex: role="button", role="navigation", role="dialog").
  • aria-label: Define uma string de texto para rotular um elemento, sendo crucial quando o conteúdo visual é insuficiente, como em botões de ícone.
  • aria-labelledby: Associa um elemento a outro que o rotula, usando o ID do elemento de rótulo.
  • aria-hidden: Oculta elementos de tecnologias assistivas (como leitores de tela), mesmo que visíveis no navegador (ex: aria-hidden="true").
  • aria-expanded: Indica se um elemento agrupador (como um menu ou acordeão) está expandido ou recolhido (true/false).
  • aria-live: Indica que um elemento será atualizado dinamicamente e os leitores de tela devem alertar o usuário (valores: polite, assertive, off).
  • aria-selected: Indica o estado de seleção de elementos, como em abas ou listas (true/false).
  • aria-checked: Indica o estado de caixas de seleção, botões de rádio ou itens de menu.
  • aria-controls: Associa um elemento ao ID do componente que ele controla ou altera.

Quais erros comuns os desenvolvedores cometem ao usar ARIA?

Os erros comuns no uso de ARIA (Accessible Rich Internet Applications) geralmente decorrem de tentativas de adicionar acessibilidade a componentes complexos sem usar as práticas recomendadas de HTML. Aqui estão os principais erros, com base em diretrizes de desenvolvimento web:

  1. Violação da "Primeira Regra do ARIA"

    2. O erro mais frequente é usar ARIA quando existe um elemento HTML nativo que já resolve o problema.

    • O erro: Criar um botão usando div ou span com role="button" e tabindex="0".
    • A correção: Use a tag button. Elementos nativos já possuem suporte a teclado (Enter/Espaço) e foco automático.

  2. Uso de aria-hidden="true" em elementos focáveis

    3. Muitos desenvolvedores ocultam elementos visualmente, mas esquecem de tirá-los da ordem de tabulação.

    • O erro: Usar aria-hidden="true" em um elemento que ainda pode receber foco por teclado, o que confunde usuários de leitores de tela.
    • A correção: Se o elemento não deve ser lido, ele não deve ser focável. Use tabindex="-1" ou remova o elemento do DOM.

  3. Falta de suporte a teclado (Keyboard Support)

    4. Atribuir um papel (role) ARIA não adiciona comportamento funcional.

    • O erro: Adicionar role="button" a um elemento div e não permitir que ele seja ativado pela tecla Enter ou Barra de Espaço.
    • A correção: Ao usar ARIA, você é responsável por gerenciar todo o comportamento via JavaScript.

  4. Rótulos genéricos ou ausentes (aria-label)

    5. Usar aria-label de forma vaga, como "clique aqui", "botão", ou "link", não dá contexto ao usuário.

    • O erro: Botões de ícones sem texto descritivo.
    • A correção: O aria-label deve ser descritivo: aria-label="Fechar janela" ou aria-label="Enviar formulário".

  5. Uso indevido de aria-live

    6. O aria-live avisa o usuário sobre mudanças na tela, mas seu uso excessivo é intrusivo.

    • O erro: Usar aria-live="assertive" (que interrompe o leitor de tela) para mensagens não críticas, como atualizações de conteúdo secundário.
    • A correção: Use aria-live="polite" para notificações gerais e aria-live="assertive" apenas para mensagens de erro urgentes.

  6. Relações quebradas com aria-labelledby

    7. aria-labelledby aponta para outro elemento para dar um nome ao componente.

    • O erro: O id referenciado no aria-labelledby não existe no DOM, ou o elemento referenciado está escondido.

  7. Uso de tabindex maior que zero

    • O erro: Usar tabindex="1", tabindex="2", etc. Isso quebra a ordem de tabulação natural da página.
    • A correção: Use apenas tabindex="0" (para colocar na ordem) ou tabindex="-1" (para remover da ordem).

Como ARIA se integra com HTML5 e CSS?

O WAI-ARIA (Accessible Rich Internet Applications) integra-se ao HTML5 e CSS agindo como uma camada semântica extra, focada em tornar aplicações web complexas acessíveis a pessoas com deficiências, especialmente usuários de leitores de tela. Ele não altera o visual (CSS) ou a funcionalidade (JavaScript), mas comunica funções, estados e propriedades aos navegadores e tecnologias assistivas.
